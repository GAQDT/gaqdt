最小时延路径

任务分析

为了获取近似的链路时延，需要通过lldp数据包获取控制器经某一交换机到另一个交换机（相邻，即一跳）再返回控制器的时延，由于这个过程是可以双向的，因此可以各进行一次获取平均值。由于每次获取的时延都包含了交换机到控制器的时延，因此需要去除这部分时延的影响。所以我们还需要利用echo数据包来获取控制器到某一交换机的往返时延。计算公式如下：
delay = (lldp_delay_s12 + lldp_delay_s21 - echo_delay_s1 - echo_delay_s2) / 2
为此，需要对Ryu进行一定修改，这里由于指导书给的比较详细，不再赘述。

容忍链路故障

任务分析

为了能够容忍链路故障，需要做到以下两点：

  • 能够捕获链路故障和故障恢复所产生的拓扑的改变
  • 能够在链路情况变化时更新流表

为了做到第一点，一定要不断地更新拓扑，可以同步也可以异步，这在必做部分已经实现；然而，为了做到第二点，必须进行流表的删除。当一条链路故障时，所有包含该链路的路径都会受到影响，为了消除这个影响，我们不妨直接删除所有受影响路径所涉及的结点（交换机）上对应的流表，这样才能够使得下一次交换机可以匹配默认流表项，向控制器发送packet_in，重新获取路径。当一条链路故障恢复时，一些原先的最小时延路径可能不再满足“最小”，因此，需要对该链路对应的两个结点进行搜索，寻找所有它们可达的主机，对于这些主机，如果任意两个之间已经有路径，则删除该路径上各结点对应的流表项。

可见，初始链路下，SDC ping MIT得到的最短时延路径是10.0.0.5 -> 1:s6:2 -> 4:s5:3 -> 3:s9:4 -> 3:s8:1 -> 10.0.0.3

此时再次尝试SDC ping MIT,得到目前的最短时延路径是10.0.0.5 -> 1:s6:3 -> 2:s7:3 -> 2:s8:1 -> 10.0.0.3

此时执行link s9 s8 up，可以发现拓扑中恢复了（8，9）边
